# Architecture Review (Read-Only) — Nabeeh

## 0) Executive Summary
- Backend code relies on teacher/password columns (`backend/routes/auth.js:53-100`) that do not exist in the published schema (`backend/database/schema_v2.sql:35-52`), so Supabase queries and RLS policies cannot succeed without manual schema drift.
- Most dashboard experiences are still demo-only: counts and cards are hardcoded (`frontend/src/app/[locale]/dashboard/page.tsx:82-95`), messaging UI uses static arrays (`frontend/src/app/[locale]/dashboard/messages/page.tsx:31-173`), and courses/schedule/report screens render local constants.
- Secrets and live session artifacts are committed to the repo (`backend/.env:5-24`, `backend/auth_info_baileys/creds.json:1`), meaning anyone cloning the project can impersonate the production WhatsApp session and Supabase instance.
- API contracts between the layers are inconsistent: e.g., the frontend sends `BulkAttendanceRequest.attendance[]` without `group_id` (`frontend/src/types/index.ts:126-133`), but the backend expects `attendance_records` with `group_id` (`backend/routes/attendance.js:83-138`); register/update-profile endpoints the UI calls simply do not exist.
- The WhatsApp automation path queries `attendance.student_id` and `grades.subject/percentage` columns that the normalized schema does not expose (`backend/routes/whatsapp.js:185-275`), so AI replies cannot be generated even if Baileys connects.
- Rate limiting and most security middleware are effectively no-ops (`backend/middleware/security.js:23-65`), while WhatsApp status/send endpoints remain unauthenticated (`backend/routes/whatsapp.js:209-237`), exposing automation controls to the public internet.
- Settings and onboarding flows are broken: the UI posts to `/api/auth/update-profile` and `/api/whatsapp/status` (`frontend/src/app/[locale]/dashboard/settings/page.tsx:116-244`) and `/api/auth/register` (`frontend/src/app/[locale]/register/page.tsx:133-156`), but those routes were never implemented.
- README instructions reference `node database/run_migration.js` and demo accounts (`README.md:25-43`) that are missing from the repo, so a clean clone cannot bootstrap the environment without digging through scripts.

## 1) Repo Map
- **Tech stack.** Next.js 15 + React 19 with the App Router (`frontend/package.json:3-33`) and Express 4 on Node 18+ (`backend/package.json:3-34`). Tailwind UI components live under `frontend/src/components/ui/`.
- **Frontend structure.** `frontend/src/app` uses locale-segmented routing (`frontend/src/app/[locale]/layout.tsx:1-23`) plus an `(auth)` group for the generic login page. Feature pages sit under `[locale]/dashboard/*`, each importing shared hooks (`frontend/src/hooks/useAuth.tsx:23-129`) and the central API client (`frontend/src/lib/api.ts:24-290`). Internationalization relies on `next-intl` (`frontend/src/messages/en.json`, `frontend/src/i18n/routing.ts:1-18`).
- **Backend structure.** `backend/server.js:1-118` wires global middleware (`backend/middleware/security.js:1-120`, `backend/middleware/logger.js:1-36`) and mounts routers from `backend/routes/*.js` (auth, teachers, students, parents, attendance, grades, messages, offerings, whatsapp). Each route module talks directly to Supabase using the shared client in `backend/config/database.js:4-7` (anon key) or, in some cases, creates its own service-role client (`backend/routes/offerings.js:6-9`).
- **Shared libs & services.** `backend/lib/auth.js:1-160` contains password/token helpers, `backend/lib/baileys.js:1-95` boots the WhatsApp socket and emits events, and `backend/scripts/*.js` are one-off Supabase utilities (`backend/scripts/seed_test_users.js:1-166`). The frontend wraps REST calls inside `frontend/src/lib/api.ts`.
- **Entrypoints & traffic flow.** The Next.js client hits Express via REST. Hard-coded base URLs (e.g., `frontend/src/lib/utils.ts:52-101`, `frontend/src/app/[locale]/dashboard/settings/page.tsx:217-244`) mean production needs reverse proxies or environment overrides. No Next.js API routes exist; everything defers to `/api/*` on Express.
- **Configuration.** `.env` files exist for both tiers, but the committed backend `.env` already contains live Supabase service keys and Gemini API keys (`backend/.env:5-24`), and WhatsApp session files are stored in `backend/auth_info_baileys/`. RLS policies are scripted in `backend/database/enable_rls_security.sql`, though many referenced tables aren’t defined anywhere else.

## 3) Static/Hardcoded Findings
- `frontend/src/app/[locale]/dashboard/messages/page.tsx:31-173` defines the entire conversation list and transcript as literal arrays. Nothing ever hits `/api/messages`, so unread counts and message content never reflect reality. **Next:** replace the arrays with data from `apiClient.getConversations`/`getConversationMessages` and handle loading states.
- Dashboard KPIs (`frontend/src/app/[locale]/dashboard/page.tsx:82-95`) show fixed numbers (“24 students”, “92% attendance”). **Next:** call `/teachers/dashboard` on mount, normalize its response (stats + recent items), and persist to context so cards reflect live data.
- Courses, schedule, and reports rely on in-module dummy arrays (`frontend/src/app/[locale]/dashboard/courses/page.tsx:21-74`, `.../schedule/page.tsx:1-140`, `.../reports/page.tsx:1-120`). **Next:** either hide these routes until APIs exist or stub an internal data-fetching hook so they can consume Supabase data once modeled.
- The System Monitor synthesizes backend/db status with random values and calls an unimplemented POST endpoint (`frontend/src/app/[locale]/dashboard/monitor/page.tsx:74-170`). **Next:** expose a proper `/api/system/status` route in Express and have the UI poll that single source instead of making multiple direct fetches.
- README advertises fully working admin/test accounts (`README.md:38-44`) and a migration script that isn’t in the tree (`README.md:25-29`). **Next:** either add the referenced script and seed data or update the documentation to describe the actual manual setup.

## 4) Data & Persistence
- **Current state.** Supabase is the only persistence layer. The shared client (`backend/config/database.js:4-7`) uses the anon key without any Supabase Auth session, which is incompatible with the RLS policies defined later. Some routes (offerings) spin up a service-role client instead (`backend/routes/offerings.js:6-9`), mixing privilege levels unpredictably.
- **Defined entities.** The v2 schema (`backend/database/schema_v2.sql:35-136`) only creates grade_levels, subjects, teachers (id, name, email, business_name), students (id, student_code, phone), teacher_subjects, offerings, groups, enrollments, attendance (by `enrollment_id`), assessments, grades, and parents. The add-on script (`backend/database/add_notes_to_grades.sql:1-1`) adds a `notes` column to grades.
- **Missing entities.** Code references `auth_audit_log` (`backend/routes/auth.js:85-99`), `password_reset_tokens` (`backend/routes/auth.js:356-510`), `teacher_settings` (`backend/routes/teachers.js:170-210`), `conversations`/`messages` (`backend/routes/messages.js:1-90`, `backend/routes/whatsapp.js:120-173`), `scheduled_messages`, `faqs`, and `documents` (`backend/database/enable_rls_security.sql:45-57`), yet none of these tables exist in `schema_v2*.sql`.
- **“Should talk to DB but doesn’t.”** The entire frontend settings flow writes to `/api/auth/update-profile` instead of the existing `/teachers/settings`, so user profile data never persists (`frontend/src/app/[locale]/dashboard/settings/page.tsx:217-244`). Dashboard/widgets/seeding rely on mocks rather than calling `apiClient` methods. WhatsApp AI responses query `attendance`/`grades` by `student_id` (`backend/routes/whatsapp.js:185-275`) despite those tables being keyed by `enrollment_id`/`assessment_id`, so Supabase always returns empty sets.

## 5) API / Contract Review
- **Missing endpoints.** The UI issues `POST /api/auth/register` and `PUT /api/auth/profile` (`frontend/src/lib/api.ts:88-105`, `frontend/src/app/[locale]/register/page.tsx:133-156`), and the settings page calls `/api/auth/update-profile` and `POST /api/whatsapp/status` (`frontend/src/app/[locale]/dashboard/settings/page.tsx:116-244`), but none of these routes exist. Requests will 404.
- **Shape mismatches.** The client believes `/attendance` returns a `PaginatedResponse<Attendance>` (`frontend/src/lib/api.ts:144-155`) and sends `BulkAttendanceRequest.attendance[]` without `group_id` (`frontend/src/types/index.ts:126-133`), whereas the backend responds with `{success,data}` only and expects `attendance_records` items that include `group_id` (`backend/routes/attendance.js:83-140`).
- **Grades contract.** The UI needs `Grade.subject`, `Grade.max_score`, etc. (`frontend/src/types/index.ts:136-175`), but `backend/routes/grades.js:203-238` only returns raw Supabase rows unless the frontend re-flattens them. The WhatsApp bot expects `grades.subject` columns that do not exist.
- **Dashboard API.** `apiClient.getDashboardStats` treats `/teachers/dashboard` as returning `DashboardStats` (`frontend/src/lib/api.ts:240-243`), yet the route actually returns `{stats, recent_grades, recent_messages}` (`backend/routes/teachers.js:72-147`). Without transformation, consumers receive the wrong shape.
- **WhatsApp endpoints.** `frontend/src/lib/utils.ts:52-101` and the monitor page (`frontend/src/app/[locale]/dashboard/monitor/page.tsx:116-158`) call hard-coded `http://localhost:5000/api/whatsapp/status` (GET and POST). The backend exposes only a GET with no auth (`backend/routes/whatsapp.js:274-288`), so POST calls fail and the GET endpoint is publicly exploitable.
- **Error handling / pagination.** Several routers use `.select(...).range(...)` but never return `count` (`backend/routes/students.js:20-67`), while the API client assumes `pagination.total` exists. Error messages still reference Mongoose in the global handler (`backend/middleware/errorHandler.js:7-34`), which is confusing when diagnosing Supabase failures.

## 6) What’s Implemented Well
- The authentication helper cleanly separates password validation, JWT issuance, and reset token generation (`backend/lib/auth.js:1-150`), making it straightforward to plug in other stores once the schema is fixed.
- Logging/security middleware is centralized: Helmet, sanitization, and request logging live in `backend/middleware/security.js:1-120` and `backend/middleware/logger.js:1-36`, so enabling/disabling enforcement only requires tweaking that layer.
- The Supabase seeding script seeds grade levels, subjects, teachers, offerings, and enrollments using the service-role key (`backend/scripts/seed_test_users.js:1-166`), which is a solid baseline for integration testing once the app consumes that data.
- On the frontend, `AuthProvider` (`frontend/src/hooks/useAuth.tsx:23-129`) and `LanguageSwitcher` (`frontend/src/components/language-switcher.tsx:1-33`) give consistent UX regardless of locale and abstract token storage away from individual screens.

## 9) Simplify & Scale Plan
- **Near-term (1–3 days).**  
  1. Rotate and remove all committed secrets and `auth_info_baileys` artifacts, then gate `/api/whatsapp/*` behind `authenticateToken`.  
  2. Ship a thin compatibility layer so `/api/attendance` accepts the payload the UI currently sends (or update the UI) and document the contract in `frontend/src/types`.  
  3. Stub `/api/auth/register`, `/api/auth/profile`, and `GET/PUT /teachers/settings` so the registration and settings pages stop failing immediately.
- **Mid-term (1–2 weeks).**  
  1. Decide whether to embrace the normalized schema or revert to the denormalized shape the UI expects; implement views or Supabase RPCs accordingly so `students`, `attendance`, `grades`, and WhatsApp automation all talk to the same shapes.  
  2. Replace hard-coded dashboard/messaging data with API-backed hooks, and centralize environment-aware API calls (e.g., via `apiClient`) rather than scattered `fetch('http://localhost:5000', …)`.  
  3. Add the missing tables (`teacher_settings`, `auth_audit_log`, `password_reset_tokens`, `conversations`, `messages`, `faqs`) via a real migration script (`database/run_migration.js`) and hook the seeding script into npm scripts.
- **Long-term (1–2 months).**  
  1. Split WhatsApp automation into its own worker/service: Express should enqueue jobs, and the worker (Baileys + Gemini) should process them with proper error handling and storage, avoiding synchronous Supabase traffic inside socket callbacks.  
  2. Introduce typed API contracts shared between Next.js and Express (e.g., via a `contracts/` package) so response shapes stay aligned as features grow.  
  3. Layer access control per domain (students, attendance, grades) and implement pagination/filter support end-to-end, ensuring Supabase policies rely on service-role keys or signed user JWTs instead of anonymous clients.

## 10) Open Questions for the Team
1. Should teachers manage their own credentials (stored in Supabase Auth) or does Nabeeh control provisioning entirely (`backend/routes/auth.js:53-150` suggests custom hashing)?
2. Are students unique per teacher or global entities shared across offerings (`backend/routes/students.js:83-205` currently enforces per-teacher enrollment only)?
3. Do parents ever interact directly with the web app, or is WhatsApp the sole channel (`backend/routes/whatsapp.js:120-275`)?
4. What data retention rules apply to WhatsApp messages and AI responses? There’s no schema for archiving beyond the undefined `messages` table.
5. Should dashboard KPIs reflect a specific time range or simply totals? The backend returns both `stats` and `recent_*` but the UI ignores them (`frontend/src/app/[locale]/dashboard/page.tsx:82-129`).
6. Are admin routes (`/${locale}/dashboard/admin/*`) supposed to exist, or should those menu items be hidden until implemented?
7. What is the expected availability/performance for WhatsApp replies—do we need retries, queues, or rate limits beyond Baileys defaults?
8. How should teacher settings be structured? The UI tracks bio, business_name, timezone, and messaging toggles (`frontend/src/app/[locale]/dashboard/settings/page.tsx:200-244`), but backend persistence is undefined.
9. Should attendance be marked per offering, per group, or per student? Current schema ties it to enrollments, but the UI doesn’t expose group selection by default.
10. Are there compliance requirements (FERPA/GDPR) affecting what can be sent over WhatsApp or stored in Supabase?
